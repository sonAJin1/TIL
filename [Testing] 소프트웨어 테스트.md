V 모델
=
V 모델은 개발 및 테스트 단계를 기준으로 개발 모델 상의 구현을 수행하는 좌측의 검증 영역과, 개발이 된 시스템을 가지고 테스트를 하는 검수 영역으로 나누어진다.

![V 모델](https://t1.daumcdn.net/cfile/tistory/99DCF2335A2E034323.jpg)

좌측 : 검증 (Right Thing)
우측 : 검수 (Right Way)

검증작업을 하기 위해서는 제품을 개발하기 전에 요구사항이나 디자인이 옳은지 검증을 하게 되는데 이러한 작업은 주로 문서를 기반으로 이루어진다. 이렇게 실제로 작동하는 시스템이 아닌 문서와 같이 정적인 것을 가지고 검증을 하는 것을 정적 테스트라고 한다. **소스 코드, 설계 문서, 요구 사항 정의서**와 같은 것을 이용한다.

우측의 동적 테스트는 작동이 가능한 실제 시스템을 기반으로 테스트를 수행한다. 각 테스트 단계에서는 좌측영역에 대응되는 각 단계의 내용을 기반으로 하여 검증을 수행한다.
단위 테스트 단계에서는 구현에 대한 신뢰성 검증을, 통합 테스트는 상세 설계에 대한 유효성 검증을, 시스템 테스트 단계에서는 설계에 대한 검증, 그리고 인수 테스트 단계에서는 사용자 요구 사항에 대한 만족도를 중심으로 테스트를 수행하게 된다.

정적 테스트
=
소스 코드를 대상으로 정적 테스트를 수행할 수 있는데, 크게 두가지 방법을 사용한다.

**코드 리뷰:**
사람이 직접 소스코드를 보면서 코드의 품질을 체크하는 프로세스이다. 비용대비 효과가 매우 큰 방식이다.
******

**정적 분석(시스템으로 자동화):**
코드 리뷰과 사람에 의해서 수동적으로 수행된다면, 정적분석은 자동화된 도구에 의해서 이루어지는 정적 분석 과정이다. 정적 분석 도구들은 소스 코드를 분석하여 다음과 같은 작업을 수행한다

 - 잘 알려진 버그에 대해서 리포팅 (ex. Connection Leak)
 - 코딩 표준에 맞지 않는 코딩 내용을 찾아냄 (띄어쓰기, 줄 바꾸기, 변수명이나 클래스명 규칙 등)
 - 실행되지 않는 데드 코드 (Dead Code) 등을 검출
 - 코드의 복잡도를 계산
 
동적 테스트
=
V 모델에 따르면 동적 테스트는 단계별로 테스트의 범위와 주체에 따라서 단위 테스트, 통합 테스트, 시스템 테스트, 인수 테스트 4가지로 분리할 수 있다.

**단위 테스트 (개발자가 개발 단계에 수행):**
단위 테스트는 시스템의 소스코드 로직 등을 점검하는 단계로, 소스 코드의 클래스나 메서드 단위의 검증을 수행한다. 이는 개발 단계에서 개발자 또는 개발팀 차원에서 직접 수행을 하게 된다. 자신이 코딩한 메소드나 클래스가 제대로 작동하는지 테스트한다.

 - **지속적 통합 (CI, Continuous Integration)**
 구현이 끝난 코드는 이 단위 테스트를 통해서 검증하고, 소스 관리 시스템에 commit 하게 된다. Commit 된 소스 코드는 다른 사람이 작성한 소스 코드와 함께 다시 컴파일이 돼서 모든 단위테스트를 다시 거치게 된다. 이를 회귀 테스트 라고 한다. 이 테스트를 하는 이유는 예전 코드의 변경이 없더라도 새로운 코드가 기존 로직에 영향을 줄 수 있기 때문에, 새 코드가 기존 코드에 대해서 결함을 발생시키지 않았음을 검증하기 위해서 수행한다. **코딩 시간과 단위 테스트 구현 시간을 같이 고려하여 개발 일정을 짜고, 모듈을 개발한 개발자가 단위 테스트까지 같이 작성하는 것이 효율성이 높다.**
 

 - **테스트 주도 개발 (TDD, Test Driven Development)**
 테스트 케이스를 먼저 만들어놓고, 코드를 작성하는 형태의 테스트 주도 개발 형식이다. 이 경우 개발자가 테스트 케이스를 먼저 개발하고 그 다음에 코드를 개발하는 형태이기 때문에, 테스트를 빠짐 없이 구현할 수 있다는 장점은 있으나, 그만큼 시간이 많이 소요된다.

******
**통합 테스트 (개발팀에서 개발 단계에 수행):**
통합 테스트는 각 개발자 또는 단위 개발팀에서 개발된 **모듈들을 서로 합쳐서 상호 연동이 제대로 되는지를 검증하는 테스트이다.** 각 서비스 컴포넌트 간의 상호 연계성을 확인 해야하기 때문에 매우 중요한 테스트 단계이다.
통합 테스트는 일반적으로 개발팀의 주도로 이루어져서 수행되며, 시스템의 규모가 클 경우에는 테스트 팀의 주도로 이루어진다. 통합 테스트에서는 시스템 연동의 기능적인 면을 주로 검토하고 성능 등의 비기능적인 면은 시스템 테스트 단계에서 수행한다.
******
**시스템 테스트 (테스트팀에서 테스트 단계에 수행):**
시스템 테스트는 통합이 완료된 시스템에 대해서 기능 테스트 및 비기능 테스트를 수행한다. **시스템 테스트는 주로 부하를 주는 상황에서 수행한다.** 제품 출시 전의 몇 번의 시스템 테스트는 테스트 환경이 아닌 실 운영 환경에서 테스트를 진행한다.
